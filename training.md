# Training Guide

## Short answer

- The denoiser in this repository is a **2D model**, not a 3D model.
- Training patches are **128 x 128**, not `128 x 128 x 128`.
- A 3D seismic cube can still be used for training, but it must be converted into many **2D slices/patches**.
- You do **not** need a pre-existing noisy dataset if you follow the paper's dynamic-guided-learning idea. You do need a source of **clean seismic patches** plus a noise simulation strategy.
- This repository contains enough code to **use the released model**, **fine-tune the denoiser**, and **run the existing synthetic degradation pipeline**.
- This repository does **not** appear to contain a complete script to train the **PGGAN generator from scratch** on a new clean dataset. It assumes a pretrained generator checkpoint at `checkpoints/PGGAN.pth`.

## What the paper does

From the paper "Poststack seismic data denoising via dynamic guided learning":

- The workflow is based on **2D poststack patches** `X in R^(M x N)`.
- Clean patches are generated by a **PGGAN**.
- Noisy patches are created by applying a **degradation model** to those clean patches.
- The denoiser is trained in a **supervised** way using dynamically generated `(noisy, clean)` pairs.
- Figure 1 in the paper describes a cycle using about **1000 clean patches** and **4000 noisy patches** per cycle.

The key point is that the paper is **not** training directly on 3D cubes. It is training on **2D images/patches extracted from seismic data**.

## What this repository implements

### Existing training-related files

- `train.py`
  - Trains the 2D denoiser using synthetic noisy/clean pairs created on the fly.
  - Loads a pretrained generator from `checkpoints/PGGAN.pth`.
  - Uses `degradationOperator.degradeBatch(...)` to create noisy versions.

- `finetuning.py`
  - Fine-tunes the released denoiser with domain adaptation logic.
  - Uses a real 2D section plus style transfer.
  - Starts from `checkpoints/att_u_fine.pt` for refinement.

- `exp_III.ipynb`
  - Notebook for domain adaptation / fine-tuning.
  - Calls `parallelTrain(section, batch_size, epochs, iterations, gen_samples)` from `finetuning.py`.

- `exp_iv.ipynb`
  - Inference / patching / reconstruction style workflow, not a full training notebook.

- `degradationOperator.py`
  - Creates synthetic noisy targets from a clean 2D patch.
  - Produces several noise types: gaussian, poisson, stripes, impulse, speckle, correlated noise, blur, streak, lines, waves, and combinations.

- `models/Attention_unet.py`
  - The denoiser architecture.
  - This is a **2D U-Net-style model** using `Conv2d`, `MaxPool2d`, and 2D feature maps.

- `models/PGGAN.py`
  - Defines the generator and discriminator architectures.
  - I did not find a full training script in the repo that trains this GAN from your raw dataset.

- `models/config.py`
  - Indicates the GAN checkpoint location: `checkpoints/PGGAN.pth`.
  - Indicates the training image size starts at `128`.

## Do you have everything needed to train from scratch?

### What you have

- A released denoiser checkpoint.
- Fine-tuning code.
- Synthetic noise generation code.
- A working inference notebook.
- GAN model definitions.

### What is missing for a true full retraining from zero

- A complete **PGGAN training pipeline** from your own clean seismic dataset.
- A documented **dataset preparation pipeline** for building the clean patch distribution used by the GAN.
- A direct supervised dataloader for user-supplied paired `(noisy, clean)` files.

### Practical conclusion

You currently have enough project code for:

1. **Inference** on new data.
2. **Fine-tuning** the denoiser using the existing checkpoints.
3. **Retraining the denoiser stage** if you still use the provided pretrained PGGAN as the clean-patch source.

You do **not** have a fully documented, end-to-end "new dataset to new PGGAN to new denoiser" pipeline in this repo.

## 2D or 3D?

The answer is:

- **Input data source** can be 3D.
- **Training samples** for this model must be **2D patches**.
- The network expects tensors shaped like `[batch, 1, 128, 128]`.

So if you have a 3D cube, the correct approach is:

1. Load the 3D cube from `.sgy`, `.segy`, or `.npy`.
2. Extract many 2D sections such as:
   - inline slices
   - crossline slices
   - optionally time/depth slices
3. Normalize each 2D section.
4. Extract overlapping or random `128 x 128` patches.
5. Use those 2D patches for training.

You should **not** try to feed `128 x 128 x 128` cubes to this repository's Attention U-Net. That would require a different **3D architecture** based on `Conv3d`.

## Recommended dataset size

There is no single exact minimum, but for a stable retraining/fine-tuning workflow I would recommend:

- Minimum useful starting point: **10,000 to 50,000** clean 2D patches of size `128 x 128`.
- Better target: **100,000+** clean patches if you want to rebuild the clean-patch distribution robustly.
- If your cube is large, you can easily reach this count by sampling from many inlines/xlines with overlap.

For your **RTX A5000**, `128 x 128` 2D patch training is entirely reasonable. The GPU is more than adequate for the denoiser stage.

## Do you need clean and noisy datasets?

There are two valid paths.

### Path A: synthetic-noise training, closest to this repo and paper

You need:

- a source of **clean seismic patches**
- the synthetic degradation pipeline to generate noisy versions on the fly

You do **not** need to store a separate noisy dataset, because noise is generated dynamically.

This is the most natural fit to the current repository.

### Path B: supervised paired training with your own real noisy-clean pairs

You need:

- a **clean** version
- the corresponding **noisy** version
- both spatially aligned patch by patch

This can work in principle, but this repository does **not** currently provide a dedicated paired-file training loader. You would need to add one.

## Can you use your own real clean 3D seismic dataset?

Yes, with an important distinction.

### If your 3D data is genuinely clean

Then it is very useful. You can:

1. Convert the cube into many 2D slices.
2. Extract many `128 x 128` clean patches.
3. Use those as the clean source.
4. Corrupt them with `degradeBatch(...)` or your own noise model.
5. Train or fine-tune the denoiser.

This is likely the best route if your goal is to adapt the model to your geology while keeping the existing workflow.

### If your "clean" real data is only relatively clean

Then be careful. If residual acquisition/processing noise is still present, the network may learn that this residual noise is part of the target signal. In that case:

- use only the cleanest subsets
- manually review patches
- reject obvious artifacts
- consider mild preprocessing before patch extraction

## Should you use only synthetic data?

No. Using only synthetic data is not mandatory.

The better answer is:

- use **real clean seismic** to define the signal distribution
- use **synthetic noise** to create broad, controllable corruption
- optionally use **real noisy sections** for fine-tuning / domain adaptation

That is also the closest match to the spirit of the paper:

- clean seismic structure should come from realistic geology
- noise variability can be broadened synthetically
- domain mismatch can be reduced by fine-tuning on target-domain examples

## Recommended workflow for your case

If you have a 3D real seismic dataset that is relatively clean and want to retrain for your own domain, this is the recommended order.

### Option 1: shortest path, most practical

1. Keep using the released `checkpoints/PGGAN.pth`.
2. Prepare one or more representative real 2D sections from your target domain.
3. Use `exp_III.ipynb` plus `finetuning.py` to fine-tune the released denoiser.
4. Validate on held-out target sections.

This is the lowest-risk route and likely the best first experiment.

### Option 2: stronger domain adaptation with your own clean data

1. Convert your clean 3D cube into many 2D `128 x 128` patches.
2. Build a patch dataset from inline/xline slices.
3. Add a new clean-patch sampling stage for the denoiser training loop.
4. Use `degradationOperator.py` to synthesize noisy inputs.
5. Train the denoiser from scratch or from `guided_att_u.pt` initialization.

This is feasible, but you will need to edit `train.py` so it reads your patches directly instead of relying only on the pretrained PGGAN generator.

### Option 3: full paper-style reproduction on your own dataset

1. Prepare a large clean patch database from your real clean 3D cubes.
2. Train a new **PGGAN** on those clean patches.
3. Replace `checkpoints/PGGAN.pth` with your generator.
4. Use `train.py` to train the denoiser with dynamic noisy-clean pairs.
5. Use `finetuning.py` or `exp_III.ipynb` for final target-domain adaptation.

This is the closest to the paper, but the repo is currently missing the full GAN-training script.

## File formats and where to load them

### Accepted formats in the inference notebook

- `.npy`
- `.sgy`
- `.segy`

The inference notebook already loads `.npy` and SEG-Y through `segyio`.

### Recommended training storage format

For training preparation, I recommend:

- keep the original seismic in `.sgy` or `.segy` if that is how you receive it
- convert to `.npy` for fast repeated patch extraction and experimentation

Why:

- `.sgy/.segy` is standard exchange format
- `.npy` is much simpler and faster for repeated training experiments in Python

### Suggested folder layout

Create something like this:

```text
datasets/
  raw/
    my_clean_cube_01.sgy
    my_clean_cube_02.sgy
  prepared/
    my_clean_cube_01.npy
    my_clean_cube_02.npy
  patches/
    train_clean_patches.npy
    val_clean_patches.npy
```

If you later implement paired training, add:

```text
datasets/
  paired/
    train_clean.npy
    train_noisy.npy
    val_clean.npy
    val_noisy.npy
```

## Concrete preparation steps

### Step 1: convert your 3D clean seismic to `.npy`

Use the loading logic from `model_usage_and_3d_denoising.ipynb` as the base. The cube should end up as:

- shape `[n_inline, n_xline, n_samples]`
- dtype `float32`

### Step 2: extract 2D slices

Use:

- inline slices: `volume[i, :, :]`
- crossline slices: `volume[:, j, :]`
- optionally timeslices: `volume[:, :, k]`

For training signal structure, inline and crossline slices are usually the safest starting point.

### Step 3: normalize and patchify

For each 2D slice:

1. normalize robustly, preferably percentile-based
2. extract `128 x 128` patches
3. use overlap or random crop sampling
4. discard empty or near-constant patches

### Step 4: choose the training regime

- If using the current repo almost unchanged: use synthetic degradation on clean patches.
- If using paired data: implement a direct paired dataloader.

### Step 5: validate on held-out slices

Hold out some entire lines or subvolumes for validation. Do not validate only on neighboring patches from the same exact section used for training.

## Existing notebooks and scripts that matter

- `train.py`: denoiser training with dynamic noisy-clean generation
- `finetuning.py`: fine-tuning / domain adaptation
- `exp_III.ipynb`: notebook entry point for fine-tuning on a real target section
- `exp_iv.ipynb`: patch extraction / reconstruction style experiment
- `model_usage_and_3d_denoising.ipynb`: inference on 2D/3D volumes and save denoised output

## Bottom line

If your goal is practical success with your own 3D seismic data, the best path is:

1. Use your real 3D cube as a source of **clean 2D patches**.
2. Keep the model **2D** and patch size **128 x 128**.
3. Start with **synthetic noise generation** using the existing degradation pipeline.
4. Fine-tune with `exp_III.ipynb` / `finetuning.py` if you have target-domain noisy examples.
5. Only attempt full from-scratch paper reproduction if you are ready to implement a **new PGGAN training pipeline**.

The most useful missing artifact in this repo is a new script such as `prepare_training_patches.py` that:

- loads `.sgy/.segy/.npy`

- normalizes them

Once that exists, retraining on your own domain becomes much more straightforward.
